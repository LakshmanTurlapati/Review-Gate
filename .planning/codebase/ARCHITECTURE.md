# Architecture

**Analysis Date:** 2026-04-02

## Pattern Overview

**Overall:** Monolithic local-integration architecture with three runtime pieces: a Cursor rule layer, a Python MCP server, and a Cursor/VS Code extension with an embedded webview UI.

**Key Characteristics:**
- The V2 runtime is split between `V2/ReviewGateV2.mdc`, `V2/review_gate_v2_mcp.py`, and `V2/cursor-extension/extension.js`; each layer owns a different stage of the review loop.
- The MCP server in `V2/review_gate_v2_mcp.py` and the extension in `V2/cursor-extension/extension.js` communicate through temp files such as `/tmp/review_gate_trigger.json`, `/tmp/review_gate_ack_<id>.json`, `/tmp/review_gate_response_<id>.json`, and `/tmp/review_gate_speech_response_<id>.json` rather than direct sockets or HTTP.
- The extension side is internally monolithic: `V2/cursor-extension/extension.js` contains activation logic, polling, popup orchestration, HTML generation, CSS, browser-side JavaScript, file upload handling, and SoX recording in one file.
- The Python side is semi-structured around the `ReviewGateServer` class in `V2/review_gate_v2_mcp.py`, but tool handlers, response polling, heartbeat logging, shutdown logic, and speech monitoring all still live in the same module.
- Installation and deployment are file-copy based. `V2/install.sh`, `V2/install.ps1`, and `V2/install.bat` install the server globally, write Cursor MCP config, and install a packaged VSIX from `V2/cursor-extension/review-gate-v2-2.7.3.vsix`.

## Layers

**Rule Layer:**
- Purpose: Instruct Cursor/agent workflows to transition into Review Gate before final completion.
- Location: `V2/ReviewGateV2.mdc`, `ReviewGate.mdc`
- Contains: Cursor rule text, workflow protocol, completion-loop requirements.
- Depends on: Cursor rule loading plus the MCP tool contract exposed by `V2/review_gate_v2_mcp.py`.
- Used by: Cursor agent sessions that load the rule file.

**MCP Server Layer:**
- Purpose: Expose Review Gate tooling to Cursor via MCP stdio and translate tool calls into popup triggers and response polling.
- Location: `V2/review_gate_v2_mcp.py`
- Contains: `ReviewGateServer`, tool registration, trigger creation, acknowledgement waiting, response collection, attachment handling, heartbeat logging, shutdown cleanup, speech monitoring, and Whisper transcription.
- Depends on: `mcp`, `faster-whisper`, local filesystem temp paths, and the Cursor extension runtime in `V2/cursor-extension/extension.js`.
- Used by: Cursor MCP process launch configured by `V2/mcp.json` or generated by `V2/install.sh`.

**Extension Host Layer:**
- Purpose: Run inside Cursor/VS Code, watch for trigger files, open the Review Gate popup, collect user input, and write response files for the server.
- Location: `V2/cursor-extension/package.json`, `V2/cursor-extension/extension.js`
- Contains: extension activation, command registration, polling loops, output-channel logging, popup creation, file/image selection, SoX recording, and cleanup.
- Depends on: VS Code extension APIs, Node built-ins (`fs`, `path`, `os`, `child_process`), SoX, and the temp-file contract used by `V2/review_gate_v2_mcp.py`.
- Used by: Cursor startup (`onStartupFinished`) and the command `reviewGate.openChat` declared in `V2/cursor-extension/package.json`.

**Embedded Webview UI Layer:**
- Purpose: Provide the actual human-facing review interface inside Cursor.
- Location: `V2/cursor-extension/extension.js`
- Contains: the HTML template returned by `getReviewGateHTML()`, inline CSS, DOM event listeners, message rendering, image previews, drag-and-drop and paste handling, and speech UI state.
- Depends on: `acquireVsCodeApi()` plus messages sent from the extension host in `V2/cursor-extension/extension.js`.
- Used by: `openReviewGatePopup()` in `V2/cursor-extension/extension.js`.

**Packaging and Installation Layer:**
- Purpose: Distribute and install the MCP server, extension, and sample config outside the repo.
- Location: `V2/install.sh`, `V2/install.ps1`, `V2/install.bat`, `V2/uninstall.sh`, `V2/uninstall.ps1`, `V2/uninstall.bat`, `V2/mcp.json`, `V2/review-gate-v2-2.7.3.vsix`, `V2/cursor-extension/review-gate-v2-2.7.3.vsix`
- Contains: installers, uninstallers, sample MCP config, packaged VSIX artifacts.
- Depends on: OS shell/PowerShell/CMD tooling, Python, pip, SoX, and optionally the `cursor` CLI.
- Used by: end-user installation and local packaging/distribution.

## Data Flow

**Agent Review Request Flow:**

1. Cursor loads the rule in `V2/ReviewGateV2.mdc`, which tells the agent to call `review_gate_chat`.
2. The MCP server in `V2/review_gate_v2_mcp.py` exposes `review_gate_chat` from `list_tools()` and handles the call in `_handle_review_gate_chat()`.
3. `_trigger_cursor_popup_immediately()` in `V2/review_gate_v2_mcp.py` writes `/tmp/review_gate_trigger.json` plus backup trigger files such as `/tmp/review_gate_trigger_0.json`.
4. `startReviewGateIntegration()` and `checkTriggerFile()` in `V2/cursor-extension/extension.js` poll those temp files, validate the payload, and forward the tool data to `handleReviewGateToolCall()`.
5. `openReviewGatePopup()` in `V2/cursor-extension/extension.js` creates or reuses a single webview panel and binds webview-to-extension message handling.
6. The extension writes `/tmp/review_gate_ack_<id>.json`; `_wait_for_extension_acknowledgement()` in `V2/review_gate_v2_mcp.py` uses that file to confirm popup activation.
7. When the user sends text or images, `logUserInput()` in `V2/cursor-extension/extension.js` writes `/tmp/review_gate_response_<id>.json`, `/tmp/review_gate_response.json`, `/tmp/mcp_response_<id>.json`, and `/tmp/mcp_response.json`.
8. `_wait_for_user_input()` in `V2/review_gate_v2_mcp.py` polls those response files, reconstructs text plus attachments, and returns them to Cursor as `TextContent` and optional `ImageContent`.

**Manual Popup Flow:**

1. Cursor activates the extension from `V2/cursor-extension/package.json` on startup or on the `reviewGate.openChat` command.
2. `activate()` in `V2/cursor-extension/extension.js` registers `reviewGate.openChat`.
3. Running the command calls `openReviewGatePopup()` with a default welcome message.
4. The webview created from `getReviewGateHTML()` handles message composition locally and sends `send`, `uploadImage`, `startRecording`, and `stopRecording` commands back to the extension host.
5. `handleReviewMessage()` in `V2/cursor-extension/extension.js` acknowledges the review inside the popup; no MCP round-trip is required for this manual path.

**Speech-to-Text Flow:**

1. The webview script embedded in `V2/cursor-extension/extension.js` toggles recording state and posts `startRecording` and `stopRecording` messages.
2. `startNodeRecording()` in `V2/cursor-extension/extension.js` validates SoX, spawns `sox`, and records to `/tmp/review_gate_audio_<id>_<timestamp>.wav`.
3. `stopNodeRecording()` stops SoX and passes the file path to `handleSpeechToText()` in `V2/cursor-extension/extension.js`.
4. `handleSpeechToText()` writes `/tmp/review_gate_speech_trigger_<id>.json`.
5. The background thread started by `_start_speech_monitoring()` in `V2/review_gate_v2_mcp.py` watches for those speech trigger files and processes them in `_process_speech_request()`.
6. `_process_speech_request()` transcribes audio with Faster-Whisper and writes `/tmp/review_gate_speech_response_<id>.json`.
7. The extension reads that response indirectly through its polling flow and forwards a `speechTranscribed` message back into the webview, which injects text into the input box.

**MCP Status Flow:**

1. `_heartbeat_logger()` in `V2/review_gate_v2_mcp.py` writes regular log lines to `/tmp/review_gate_v2.log`.
2. `startMcpStatusMonitoring()` and `checkMcpStatus()` in `V2/cursor-extension/extension.js` check the modification time of `/tmp/review_gate_v2.log`.
3. `updateChatPanelStatus()` sends `updateMcpStatus` into the webview so the UI can flip between active and inactive states.

**State Management:**
- Extension-host state is global mutable module state in `V2/cursor-extension/extension.js`: `chatPanel`, `reviewGateWatcher`, `outputChannel`, `mcpStatus`, `statusCheckInterval`, `currentTriggerData`, and `currentRecording`.
- Webview state is browser-local mutable state inside the template returned from `getReviewGateHTML()` in `V2/cursor-extension/extension.js`: `attachedImages`, `isRecording`, `mcpIntegration`, and textarea/UI state.
- MCP server state is instance state on `ReviewGateServer` in `V2/review_gate_v2_mcp.py`: `shutdown_requested`, `shutdown_reason`, `_last_attachments`, `_whisper_model`, `_whisper_error`, `_speech_monitoring_active`, and `_speech_thread`.
- Persistent state is not database-backed. Long-lived coordination uses temp files under `/tmp` or `tempfile.gettempdir()` plus Cursor config written outside the repo by `V2/install.sh`.

## Key Abstractions

**Review Gate Trigger Envelope:**
- Purpose: Represent a request from the MCP server to the extension.
- Examples: `V2/review_gate_v2_mcp.py`, `V2/cursor-extension/extension.js`
- Pattern: JSON files such as `/tmp/review_gate_trigger.json` containing `system`, `editor`, `data`, `pid`, and `immediate_activation`.

**Single Popup Singleton:**
- Purpose: Ensure only one Review Gate panel exists at a time in Cursor.
- Examples: `V2/cursor-extension/extension.js`
- Pattern: `chatPanel` is a module-level singleton reused by `openReviewGatePopup()` instead of creating multiple panels.

**Webview Message Protocol:**
- Purpose: Carry user actions between the embedded popup and the extension host.
- Examples: `V2/cursor-extension/extension.js`
- Pattern: String command messages such as `send`, `uploadImage`, `startRecording`, `stopRecording`, `updateMcpStatus`, `imageUploaded`, and `speechTranscribed`.

**Response Envelope with Attachments:**
- Purpose: Return human text plus optional images from the extension back to the MCP server.
- Examples: `V2/cursor-extension/extension.js`, `V2/review_gate_v2_mcp.py`
- Pattern: JSON response files containing `trigger_id`, `user_input`, `attachments`, `event_type`, and `source`, later converted into `TextContent` and `ImageContent`.

**Speech Monitor Side Channel:**
- Purpose: Handle speech transcription separately from the main MCP stdio request loop.
- Examples: `V2/review_gate_v2_mcp.py`, `V2/cursor-extension/extension.js`
- Pattern: background thread in `V2/review_gate_v2_mcp.py` plus SoX-spawned audio capture in `V2/cursor-extension/extension.js`, coordinated with speech trigger/response temp files.

## Entry Points

**Cursor Extension Activation:**
- Location: `V2/cursor-extension/package.json`, `V2/cursor-extension/extension.js`
- Triggers: `onStartupFinished` and `onCommand:reviewGate.openChat`
- Responsibilities: load the extension host, register the command, start MCP status monitoring, and start polling for Review Gate trigger files.

**Manual Review Gate Command:**
- Location: `V2/cursor-extension/package.json`
- Triggers: `Cmd+Shift+R` on macOS and `Ctrl+Shift+R` on Windows/Linux
- Responsibilities: open the popup without an MCP server round-trip.

**MCP Server Startup:**
- Location: `V2/review_gate_v2_mcp.py`
- Triggers: Cursor launching the configured MCP command from `V2/mcp.json` or the installation-generated `~/.cursor/mcp.json`
- Responsibilities: initialize `ReviewGateServer`, register tools, start heartbeat and shutdown tasks, and keep the stdio server alive.

**Installer Entry Points:**
- Location: `V2/install.sh`, `V2/install.ps1`, `V2/install.bat`
- Triggers: manual user installation
- Responsibilities: create a global install directory, create a virtual environment, install dependencies, configure Cursor MCP, and install/copy the VSIX.

## Error Handling

**Strategy:** Best-effort local recovery with logging, timeouts, and user-visible fallbacks instead of strict exception propagation.

**Patterns:**
- `V2/review_gate_v2_mcp.py` wraps most tool and polling logic in `try/except` blocks and returns `TIMEOUT:` or `ERROR:` text payloads to the MCP caller rather than crashing the server.
- `V2/cursor-extension/extension.js` catches file-system and JSON errors while polling trigger/response files so transient temp-file failures do not terminate the extension.
- Speech handling validates dependencies before recording. `validateSoxSetup()` in `V2/cursor-extension/extension.js` surfaces microphone and SoX problems to the popup UI.
- Cleanup is eager and file-based. Both `V2/review_gate_v2_mcp.py` and `V2/cursor-extension/extension.js` remove trigger, acknowledgement, response, and audio files where possible after processing.

## Cross-Cutting Concerns

**Logging:** `V2/review_gate_v2_mcp.py` writes to `/tmp/review_gate_v2.log` and stderr; `V2/cursor-extension/extension.js` logs to the console, a VS Code output channel, and `/tmp/review_gate_user_inputs.log`.

**Validation:** `V2/cursor-extension/extension.js` validates `editor` and `system` fields before accepting a trigger, validates SoX availability before recording, and validates file existence before image/audio processing. `V2/review_gate_v2_mcp.py` validates trigger/response JSON and trigger IDs before accepting responses.

**Authentication:** None inside the repo. Trust is local-process based: Cursor starts the MCP server from `V2/mcp.json` or installed config, and the extension accepts temp-file messages that identify `system` as `review-gate-v2`.

**Concurrency:** The MCP server in `V2/review_gate_v2_mcp.py` is async for tool handling but also runs a background speech-monitor thread. The extension in `V2/cursor-extension/extension.js` uses timer-based polling for trigger/status files and process-level concurrency for SoX recording.

**Module Boundaries:** The strongest boundary is across files, not within them: `V2/review_gate_v2_mcp.py` owns server/tool orchestration and `V2/cursor-extension/extension.js` owns IDE/UI orchestration. Inside each file, boundaries are functional clusters rather than reusable modules.

**Runtime Surface Area:** `list_tools()` in `V2/review_gate_v2_mcp.py` currently exposes only `review_gate_chat`. The same file also contains handlers for unified, quick, file, ingest, and shutdown flows, but they are internal unless additional tool registrations are added.
